🚧 Podman + Rootless + SELinux
===

При развертывании в энтерпрайз-дистрибутивах стандартная конфигурация «из коробки» гарантированно столкнется с ограничениями
безопасности ядра. Ниже описаны основные проблемы и способы их решения.

> ⚠️ **ДИСКЛЕЙМЕР:** Этот материал описывает базовые ограничения среды и способы их решения «в лоб» ради быстрого запуска. Применяя
> данные инструкции, вы действуете на свой страх и риск, осознавая последствия для безопасности системы.

### Типичные грабли rootless-режима

* **Permission Denied на сокет/тома:** SELinux блокирует доступ контейнера к хостовым путям, даже если права `ls -l` выставлены в
  `777`. Наличие механизма `subuid/subgid` в rootless-режиме подменяет UID (пользователь `traefik` внутри контейнера — это не
  UID `1000` для ядра хоста).
* **Смерть контейнера при выходе из SSH:** По умолчанию systemd убивает все процессы rootless-пользователя, как только закрывается
  терминальная сессия.
* **Порты 80 и 443 заблокированы:** Непривилегированный пользователь по умолчанию не может биндить порты ниже `1024`.

## 🛠 Путь компромиссов (Быстрый запуск «как на десктопе»)

Если вам нужно развернуть стек быстро в тестовой/домашней среде, выполните следующие шаги.

> 🚨 **РИСКИ:** Полное отключение SELinux снижает изоляцию хоста. Если контейнер будет скомпрометирован, злоумышленнику будет
> значительно проще совершить побег в ОС. Открытие привилегированных портов для rootless-юзеров теоретически позволяет любому
> локальному скрипту занять веб-порты хоста.

**Шаг 1. Отключение SELinux (решает проблемы с правами доступа к файлам/сокетам):**

```bash
sudo sed -i 's/SELINUX=enforcing/SELINUX=disabled/g' /etc/selinux/config
sudo reboot
```

**Шаг 2. Разрешение бинда портов 80 и 443 для rootless-пользователей:**

По умолчанию лимит равен 1024. Опустим его до 0:

```bash
sudo sysctl -w net.ipv4.ip_unprivileged_port_start=0
# Сделаем настройку персистентной:
echo "net.ipv4.ip_unprivileged_port_start=0" | sudo tee -a /etc/sysctl.d/99-rootless-ports.conf
```

**Шаг 3. Создание и запуск пользовательского Systemd-юнита без доп. пакетов:**

Podman умеет сам генерировать корректную обвязку из уже запущенного через compose контейнера.

```bash
# 1. Генерируем юнит (флаг --new заставляет его пересоздавать контейнер при старте)
mkdir -p ~/.config/systemd/user/
podman generate systemd --name traefik --files --new
mv container-traefik.service ~/.config/systemd/user/traefik.service
# 2. Активируем службу (БЕЗ sudo!)
systemctl --user daemon-reload
systemctl --user enable --now traefik.service
# 3. КРИТИЧНО: Разрешаем процессам пользователя жить после закрытия SSH-сессии (Linger)
loginctl enable-linger $USER
```

## 💡 Альтернативы и здравый смысл

### 1. Простой путь: Классический Docker (Root-mode)

Если вам нужно развернуть инфраструктуру без вышеописанного «геморроя» с контекстами безопасности, сопоставлением UID и пробросом
портов, используйте **официальный Docker Engine** вместо Podman.

* Демон Docker работает от имени `root`, поэтому порты `80/443` биндятся из коробки.
* SELinux-политики для Docker в стандартных репозиториях настроены более прозрачно и не требуют отключения всей системы защиты
  хоста ради одного контейнера.
* Достаточно установить `docker-ce`, `docker-compose-plugin` и использовать классический режим `docker-compose.socket-root.yml`.

### 2. Когда действительно нужен Rootless + Podman + RHEL

Если политика безопасности вашей компании или проекта **строго требует** изоляции на базе Rootless Podman с включенным SELinux
(режим `Enforcing`) в enterprise-окружении:

* **Не пытайтесь чинить это «костылями» и отключением защитных механизмов.**
* Обратитесь к системным администраторам или инженерам информационной безопасности (SecOps/DevSecOps).
* Правильный production-подход в данной конфигурации требует тонкой настройки политик SELinux через типы контекстов (
  `container_file_t`, `container_share_t`), ручной сборки кастомных модулей (`.pp`), конфигурирования политик `seccomp` и тонкой
  настройки пространства имен (Namespaces). Попытки решить это «в лоб» на боевых серверах приведут либо к дырам в безопасности,
  либо к нестабильной работе сети.
